# Аудит: обработка OUTPUT-параметров stored procedures

**Дата**: 2026-07-15
**Затронутые компоненты**: Weir.Core (ParameterBinder), Weir.Connectors.SqlServer, Weir.Connectors.PostgreSql
**Коммит**: cd5841e

---

## Описание проблемы

При вызове stored procedure с OUTPUT-параметром типа `nvarchar(max)` через админ-панель (Test / invoke) без указания значения параметра -- запрос падал с ошибкой. При смене параметра на обычный INPUT ошибка исчезала.

## Корневая причина

Цепочка обработки OUTPUT-параметра при отсутствии значения в запросе:

1. **Интроспекция** (`SqlServerConnector.DescribeParametersAsync`, строка 225): SQL Server маппит все output-параметры как `InputOutput` (не `Output`). Для `nvarchar(max)` значение `max_length` в `sys.parameters` равно `-1`, и `Size` устанавливается в `null` (строка 226: `r.MaxLength > 0 ? r.MaxLength : null`).

2. **Биндинг** (`ParameterBinder.BindOne`, строка 93): проверяется `definition.Direction == ParameterDirection.Output`. Поскольку интроспекция вернула `InputOutput`, условие не срабатывает и код проваливается в `ReadValue`. Если значение не передано в запросе, `ReadValue` возвращает `(false, null)`. Параметр не обязательный (`Required = false`), поэтому `Value = DefaultValue = null`.

3. **Коннектор** (`SqlServerConnector.AddParameters`, строка 403): `parameter.Value = wp.Value ?? DBNull.Value` -- `DBNull.Value` передаётся в SQL Server. Но `SqlParameter.Size` **не задан** (null -> 0 по умолчанию в SqlClient).

4. **Исполнение**: SQL Server пытается записать результат OUTPUT-параметра обратно через нулевой буфер. Для строковых типов `Size = 0` означает буфер в 0 байт -- значение обрезается или отвергается.

## Исправление

В `SqlServerConnector.AddParameters` и `PostgreSqlConnector.AddParameters` добавлена проверка: если `Size` не задан и параметр имеет направление `Output` или `InputOutput`, явно устанавливается `Size = -1` (максимальный буфер).

```csharp
else if (wp.Direction is WeirDirection.Output or WeirDirection.InputOutput)
{
    parameter.Size = -1;
}
```

Файл: `src/connectors/Weir.Connectors.SqlServer/SqlServerConnector.cs`, строки 392-397.
Файл: `src/connectors/Weir.Connectors.PostgreSql/PostgreSqlConnector.cs`, строки 303-308.

Тесты: 103/103 пройдены.

## Предложения по улучшению

### 1. Асимметрия ParameterBinder между Output и InputOutput

**Файл**: `src/Weir.Core/ParameterBinder.cs`, строки 93-106.

`BindOne` обрабатывает чистый `Output` (возврат с `Value = null`) и `InputOutput` (провал в `ReadValue`) по-разному. На практике это работает, потому что SQL Server всегда маппит output-параметры как `InputOutput`. Однако если появится коннектор, возвращающий чистый `Output`, поведение будет некорректным: `ReadValue` попытается прочитать значение из тела запроса для параметра, который не должен иметь входного значения.

**Предложение**: объединить обработку `Output` и `InputOutput` в одном блоке. Для `InputOutput` без входного значения -- не передавать `DBNull.Value`, а оставить `Value = null`, чтобы коннектор мог принять решение (например, не добавлять параметр в команду или использовать значение по умолчанию).

### 2. Буфер по умолчанию для max-типов

**Файл**: `src/connectors/Weir.Connectors.SqlServer/SqlServerConnector.cs`, строка 395.

Текущий фикс устанавливает `Size = -1` для всех OUTPUT-параметров без заданного размера. Это безопасно, но для фиксированных строковых типов (например, `nvarchar(255)`) интроспекция уже возвращает корректный `Size`, поэтому `-1` применяется только к max-типам. Тем не менее, для строковых OUTPUT-параметров с неизвестным размером можно рассмотреть более консервативный буфер (например, 4000 символов), чтобы избежать чрезмерного потребления памяти при большом количестве одновременных запросов.

**Предложение**: для `NVarChar` / `VarChar` / `VarBinary` без заданного `Size` использовать `4000` вместо `-1`. Для `Xml` и других типов, где `-1` является стандартом -- оставить `-1`.

### 3. Кеширование INPUT/OUTPUT-параметров

**Файл**: `src/Weir.Core/ParameterBinder.cs`, строка 149.

Для `InputOutput`-параметров без входного значения `Value` попадает в `values` (словарь для кеширования) как `null`. Это означает, что два запроса с разными OUTPUT-значениями (но одинаковыми INPUT-значениями) получат одинаковый кеш-ключ, хотя результаты могут различаться.

**Предложение**: исключить `InputOutput`-параметры без входного значения из кеш-ключа (`values`), либо генерировать уникальный токен для таких параметров.

### 4. Документирование OUTPUT-параметров в OpenAPI

**Файл**: `src/Weir.Host/http/OpenApiGenerator.cs`, строка 82.

Генератор OpenAPI корректно пропускает OUTPUT-параметры в схеме запроса, но не описывает их в схеме ответа. Клиенты, генерирующие код по OpenAPI-спецификации, не получат информации о структуре объекта `output` в ответе.

**Предложение**: добавить OUTPUT-параметры в схему ответа (`response schema`) как свойства объекта `output` с соответствующими типами.

### 5. UX админ-панели: визуальное разделение INPUT и OUTPUT

**Файл**: `src/Weir.Admin/` (Blazor UI).

При тестировании procedure через админ-панель все параметры отображаются в единой форме. Пользователь не видит, какие параметры являются OUTPUT и не должны заполняться.

**Предложение**: визуально отделить OUTPUT-параметры (например, пометить как "read-only" или вынести в отдельную секцию "Output parameters"), чтобы пользователь не пытался передать для них значения.
