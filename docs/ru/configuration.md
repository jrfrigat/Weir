# Weir - Конфигурация

> [English](../en/configuration.md) - [Деплой](deployment.md) - [Безопасность](security.md)

Weir читает стандартную конфигурацию ASP.NET Core: `appsettings.json`,
`appsettings.{Environment}.json`, переменные окружения и аргументы командной строки (в порядке
возрастания приоритета). Вложенные ключи маппятся на переменные окружения через двойное
подчёркивание, например `Weir:DataConnections:default:ConnectionString` становится
`Weir__DataConnections__default__ConnectionString`.

## Настройки

### `Weir:ControlPlane`

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Provider | `Sqlite` | Стор control-plane. `Sqlite` (один узел) или `Postgres` / `SqlServer` (общая серверная БД, для высокой доступности). |
| ConnectionString | `Data Source=weir-control.db` | Строка подключения к собственному стору метаданных Weir. Для `Postgres` нужна строка подключения Npgsql. |
| ReloadSeconds | `0` | Как часто каждый инстанс перечитывает каталог эндпоинтов из стора. Задайте, например, `30` в мультиинстанс (HA) развёртывании, чтобы изменения метаданных на одном инстансе дошли до других; `0` отключает (один узел). |

Для развёртывания с высокой доступностью запустите несколько инстансов Weir против одной
control-базы PostgreSQL:

```sh
Weir__ControlPlane__Provider=Postgres
Weir__ControlPlane__ConnectionString="Host=pg;Database=weir_control;Username=weir;Password=..."
Weir__HighAvailability=true
```

`Weir:HighAvailability` (по умолчанию `false`) утверждает, что развёртывание многоинстансовое. При
`true` хост отказывается стартовать на одноузловом SQLite control plane и требует общий PostgreSQL
control plane, чтобы несколько инстансов не работали молча против расходящихся пофайловых метаданных. Полный
чек-лист HA (общий ключ JWT, Redis-лимитер, cross-instance метрики через OTLP) см. в
[Развёртывание - Высокая доступность](deployment.md#высокая-доступность).

### `Weir:DataConnections:{name}`

По одной записи на именованное целевое подключение. Эндпоинты ссылаются на подключение по `{name}`.

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Provider | `SqlServer` | Провайдер data-plane: `SqlServer` или `PostgreSql`. |
| ConnectionString | (обязательно) | ADO.NET строка подключения к целевой БД. |
| DefaultCommandTimeoutSeconds | (нет) | Таймаут команды по умолчанию, когда эндпоинт его не задал. |

Оба коннектора повторяют только транзиентные ошибки открытия подключения (до выполнения команды),
поэтому процедура никогда не вызывается более одного раза.

### `Weir:Admin`

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Username | (нет) | Логин bootstrap-админа. |
| Password | (нет) | Пароль bootstrap-админа. Админ создаётся при старте только если админа нет и оба значения заданы. Не короче 8 символов (минимальная парольная политика Weir); более слабое значение логируется и пропускается. |
| MaxFailedLogins | `5` | Число неудачных входов с одного клиента (IP-адреса) до временной блокировки. Ноль отключает блокировку. |
| LockoutMinutes | `15` | Как долго заблокированный клиент остаётся заблокированным. |
| MaxTokensPerAdmin | `20` | Максимум персональных токенов у одного админа одновременно. Ноль или меньше - без ограничения. |
| RequireTokenExpiry | `false` | Если true, персональный токен обязан создаваться с истечением; токен без истечения отклоняется. |

### `Weir:Jwt`

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Issuer | `weir` | Issuer JWT. |
| Audience | `weir-admin` | Audience JWT. |
| SigningKey | (обязателен в Production) | Симметричный секрет подписи. В Production хост отказывается стартовать при пустом значении или значении короче 32 символов; в Development генерируется эфемерный ключ (сессии слетают при рестарте и не валидны между инстансами). |
| AccessTokenMinutes | `30` | Время жизни access-токена админа. Короткое, потому что refresh-токен незаметно продлевает сессию; более короткий access-токен сужает окно, в котором утёкший или ещё не отозванный токен пригоден. Токены также отзываемы до истечения: смена пароля админа или отключение аккаунта отклоняет его действующие токены при следующем запросе. |
| RefreshTokenDays | `14` | Время жизни refresh-токена админа. Refresh-токен долгоживущий, отзываемый и ротируется при каждом использовании; он обменивается (`POST /admin/api/auth/refresh`) на новый access-токен, когда access-токен истекает. Смена пароля или выход его отзывают. |

### `Weir:DataPlane`

Гарды, защищающие сервис и базу от одного тяжёлого вызова. Это seed / значения по умолчанию;
большинство также редактируется в рантайме на экране **Settings** в админке (см. [Настройки в
рантайме](#настройки-в-рантайме) ниже), который накладывает сохранённое значение на seed без перезапуска.

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| MaxRows | `100000` | Максимум строк (по всем result set) в одном ответе. При превышении ответ закрывается досрочно, помечается `"truncated": true` и не кэшируется. Ноль - без ограничения. |
| RequestTimeoutSeconds | `30` | Общий таймаут запроса; по истечении клиент получает HTTP 504. Ноль - без gateway-таймаута (таймаут команды БД всё равно действует). |
| MaxTvpRows | `100000` | Максимум строк для одного table-valued параметра; больший массив отклоняется с HTTP 400. Ноль - без ограничения. |
| MaxRequestBodyBytes | `10485760` | Максимальный размер тела запроса, проверяется веб-сервером (HTTP 413 при превышении). Применяется на старте, поэтому изменение требует перезапуска. Ноль или меньше - дефолт сервера. |
| DefaultApiKeyRateLimitPerMinute | `0` | Дефолтный лимит в минуту для ключа, у которого свой не задан. Ноль или меньше - ключ не троттлится. |
| MaxConcurrentRequestsPerConnection | `0` | Bulkhead: максимум одновременных выполнений на одно data-соединение. Запрос сверх лимита быстро отклоняется с HTTP 503. Ноль - без ограничения. |
| CircuitBreakerFailureThreshold | `0` | Число подряд идущих сбоев data-соединения, открывающих circuit breaker для этого соединения; пока он открыт, запросы коротко замыкаются на HTTP 503 до успешного пробного. Ноль отключает breaker. |
| CircuitBreakerResetSeconds | `30` | Сколько секунд сработавший breaker остаётся открытым, прежде чем пропустить пробный запрос. |
| ApiKeyFailureThreshold | `20` | Сколько неразрешённых API-ключей один адрес вызывающего может предъявить за минуту, прежде чем дальнейшим запросам откажут с HTTP 429 до обращения к базе. Ограничивает поток случайных ключей, который иначе шёл бы в хранилище на каждый запрос (неизвестный ключ не кэшируется никогда). Валидный ключ против него не считается. Ноль отключает гард. На адрес вызывающего, поэтому за прокси требует `Weir:Network:TrustedProxies`. |
| ResponseCacheMaxBytes | `134217728` | Сколько всего байт памяти могут занимать кэшированные ответы. При заполнении вытесняются самые давно использованные записи; ответ больше лимита не кэшируется никогда. Ноль или меньше - без ограничения: тогда эндпоинт с высокой кардинальностью `VaryByParameters` может растить кэш, пока в процессе не кончится память. Изменение пересоздаёт кэш, отбрасывая накопленные записи. |

#### Настройки в рантайме

`MaxRows`, `RequestTimeoutSeconds`, `MaxTvpRows`, `DefaultApiKeyRateLimitPerMinute`,
`MaxConcurrentRequestsPerConnection`, `CircuitBreakerFailureThreshold`, `CircuitBreakerResetSeconds`,
`ApiKeyFailureThreshold` и `ResponseCacheMaxBytes` редактируются в рантайме на экране **Settings** в
админке (или `GET` / `PUT /admin/api/settings`, роль Admin, с аудитом).
Изменённые значения хранятся в control-plane, поэтому переживают перезапуск и доходят до всех инстансов,
которые делят общую control-базу. `MaxRequestBodyBytes` показан там только для чтения, так как
применяется к веб-серверу на старте и требует перезапуска.

Ещё три настройки существуют только в рантайме - у них нет seed в `appsettings.json`, и редактируются
они там же, в группе **Request log**:

| Настройка | По умолчанию | Смысл |
| :-- | :-- | :-- |
| RequestLogEnabled | `true` | Главный переключатель лога запросов data-plane (история по вызовам, показываемая в админке). При выключении строки лога запросов не пишутся независимо от настроек эндпоинтов. При включении пишутся базовые метаданные вызова; захват параметров и результата остаётся opt-in на эндпоинт. |
| SlowRequestThresholdPercent | `20` | Дефолтный порог "медленного": процент сверх скользящего среднего эндпоинта - залогированный запрос помечается медленным, когда его длительность превышает недавнее среднее эндпоинта хотя бы на столько (например, `20` значит "на 20% медленнее обычного"). Эндпоинт может переопределить. Ноль или меньше отключает пометку медленных. |
| RequestLogRetentionDays | `7` | Сколько дней хранить историю лога запросов; более старые строки чистит фоновый сервис. Ноль или меньше - хранить вечно (не рекомендуется: лог растёт быстро). |

### `Weir:Security`

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| ContentSecurityPolicy | (политика, совместимая с Blazor) | Значение заголовка `Content-Security-Policy`. Пустая строка - не отправлять. |
| RequireHttps | `false` | Редирект plain HTTP на HTTPS внутри процесса. Оставьте выключенным, если TLS терминирует reverse proxy / ingress. |

### `Weir:RateLimit`

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| RedisConnectionString | (пусто) | Если задан (например `localhost:6379`), per-API-key rate limiter считает запросы в Redis, поэтому один лимит действует на все инстансы, вместо in-memory лимитера по умолчанию, который применяет лимит на инстанс (N x лимит в HA из N инстансов). При кратковременной недоступности Redis лимитер fail-open (пропускает запрос) и пишет предупреждение. |
| RedisKeyPrefix | `weir:ratelimit:` | Префикс ключей лимитера в Redis, чтобы делить Redis с другими приложениями. |

### `Weir:Plugins`

Загрузка плагинов (например сторонних коннекторов) в готовый образ без пересборки. См.
[Расширение](extending.md). Плагины выполняются в процессе - загружайте только доверенные сборки.

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Paths | (пусто) | Пути к сборкам плагинов для загрузки при старте, например `Weir__Plugins__Paths__0=/plugins/Acme.Weir.MySql.dll`. |

Weir также всегда шлёт `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY` и
`Referrer-Policy: no-referrer`, и `Strict-Transport-Security` (HSTS) вне Development.

## Стандартные настройки хоста

| Ключ / переменная | Смысл |
| :-- | :-- |
| ASPNETCORE_URLS | Адрес(а) прослушивания, например `http://+:8080`. |
| ASPNETCORE_ENVIRONMENT | `Development` / `Production`. Development подгружает `appsettings.Development.json`. |
| Logging:LogLevel:* | Стандартная настройка логирования. |

## Пример (переменные окружения)

```sh
ASPNETCORE_URLS=http://+:8080
Weir__ControlPlane__ConnectionString="Data Source=/data/weir-control.db"
Weir__DataConnections__default__Provider=SqlServer
Weir__DataConnections__default__ConnectionString="Server=mssql;Database=Demo;User Id=sa;Password=...;TrustServerCertificate=True"
Weir__Admin__Username=admin
Weir__Admin__Password=надёжный-пароль
Weir__Jwt__SigningKey=стабильный-секрет
```

## CORS

Браузерным клиентам, которые вызывают data-plane API с другого origin, нужен CORS. CORS включается,
только когда список разрешённых origin непустой.

| Ключ | Смысл |
| :-- | :-- |
| Weir:Cors:AllowedOrigins | Массив разрешённых origin, например `[ "https://app.example.com" ]`. |

## Rate limiting

У каждого API-ключа может быть `RateLimitPerMinute` (задаётся в админке). Запросы сверх лимита
получают HTTP 429 с заголовком `Retry-After`. По умолчанию лимитер работает на инстанс (in-memory);
задайте [`Weir:RateLimit:RedisConnectionString`](#weirratelimit), чтобы считать в Redis, и тогда один
лимит действует на все инстансы мультиинстансового развёртывания.

## Аудит

Административные действия (входы, изменения ключей) аудируются всегда. Аудит вызовов data-plane
включается опционально, так как пишет по одной строке control-plane на каждый запрос; записи
ставятся в очередь и пишутся фоновым writer-ом, поэтому они не блокируют горячий путь запроса.

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Weir:Audit:DataPlane | `false` | Записывать по одной аудит-записи на каждый вызов data-plane. |
| Weir:Audit:QueueCapacity | `10000` | Граница in-memory очереди аудита; при переполнении записи отбрасываются, чтобы запросы не блокировались. |

История аудита чистится фоновым сервисом по настройке рантайма `AuditRetentionDays` (редактируется на
экране **Settings** в админке; ноль - хранить вечно). Когда очередь аудита отбрасывает записи под
нагрузкой, в лог пишется предупреждение с накопленным числом потерь.

## Логирование

Weir логирует через Serilog. Он пишет rolling-файл (и, опционально, консоль); каждый запрос эмитит одну
структурированную сводную строку с correlation id (заголовок запроса `X-Correlation-ID` или
сгенерированное значение, возвращаемое в ответе). Логируются события безопасности - неудачные входы,
блокировки, отказы по scope/grant, срабатывания rate-limit и изменения настроек. Конфигурация файла также
показана только для чтения на экране **Settings** в админке.

| Ключ | По умолчанию | Смысл |
| :-- | :-- | :-- |
| Weir:Logging:FileEnabled | `true` | Писать логи в rolling-файл. |
| Weir:Logging:ConsoleEnabled | `true` | Также писать логи в консоль. |
| Weir:Logging:Directory | `logs` | Директория для файлов логов (создаётся при отсутствии; относительно content root). |
| Weir:Logging:FileName | `weir-.log` | Базовое имя файла; дата ротации и, при size-роле, порядковый номер вставляются перед расширением. |
| Weir:Logging:RollingInterval | `Day` | `Infinite`, `Year`, `Month`, `Day`, `Hour` или `Minute`. |
| Weir:Logging:FileSizeLimitBytes | `52428800` | Ролл в новый файл при достижении этого размера. Null отключает size-ролл. |
| Weir:Logging:RetainedFileCountLimit | `31` | Максимум хранимых rolling-файлов. Null - хранить все. |
| Weir:Logging:RetainedFileTimeLimitDays | (нет) | Максимальный возраст хранимых файлов в днях. Null отключает лимит по времени. |
| Weir:Logging:Format | `Text` | `Text` (человекочитаемый) или `Json` (compact JSON, один объект на строку; включает correlation id). |
| Weir:Logging:MinimumLevel | `Information` | `Verbose`, `Debug`, `Information`, `Warning`, `Error` или `Fatal`. |

Настройки логирования применяются на старте, поэтому изменение требует перезапуска.

## Телеметрия

Weir эмитит метрики и трейсы на meter и activity source `Weir` (плюс инструментация ASP.NET Core) и
держит in-memory агрегатор, который питает встроенный дашборд без внешнего бэкенда. Задайте
`OTEL_EXPORTER_OTLP_ENDPOINT`, чтобы экспортировать метрики и трейсы по OTLP в любой
OpenTelemetry-коллектор или дашборд .NET Aspire.
