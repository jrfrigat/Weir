# Weir - Безопасность

> [English](../en/security.md) - [Конфигурация](configuration.md) - [Архитектура](architecture.md)

У Weir две независимые поверхности аутентификации: API-ключи для клиентов data-plane и аккаунты
админов для интерфейса управления.

## Клиенты data-plane: API-ключи

- Клиент аутентифицируется API-ключом в заголовке `X-Api-Key` или как `Bearer`-токен.
- Ключи выглядят как `wk_live_<random>`. Хранятся только SHA-256-хеш и короткий несекретный префикс;
  plaintext показывается один раз при создании и не сохраняется.
- Каждый ключ несёт набор скоупов. Эндпоинт объявляет `RequiredScopes`; запрос допускается, только
  если ключ обладает всеми ими. Эндпоинт без требуемых скоупов принимает любой включённый ключ.
- Ключи можно отключать (отзывать) и задавать им опциональный срок действия.
- Резолвнутые ключи кратко кешируются в памяти, чтобы горячий путь не бил в БД.

Создавайте и отзывайте ключи в разделе **API keys** админки или через admin API
(`POST /admin/api/keys`, `DELETE /admin/api/keys/{id}`).

## Скоупы

Скоупы - это именованные права (например `orders:read`, `orders:write`). Определите их в разделе
**Scopes**, привяжите к ключам и требуйте на эндпоинтах. Неявной иерархии нет - скоуп требуется ровно
тогда, когда эндпоинт его перечисляет.

## Доступ к процедурам (resource grants)

Помимо скоупов, ключ может нести **resource grants** - права, ограничивающие, какие процедуры он
может вызывать, по уровням: подключение (именованное подключение - это один сервер + база данных),
схема и объект. Каждый уровень - это конкретное значение или `*` ("любой"), так что один grant может
покрывать:

| Grant | Смысл |
| :-- | :-- |
| `* / * / *` | любая процедура на любом подключении |
| `sales / * / *` | все процедуры на подключении `sales` (весь его сервер + база) |
| `sales / dbo / *` | все процедуры в схеме `dbo` подключения `sales` |
| `sales / dbo / GetOrders` | только эта одна процедура |

У ключа может быть много grant'ов; вызов разрешён, если хотя бы один grant совпадает с подключением,
схемой и объектом эндпоинта. Ключ **без** grant'ов не ограничен (но всё ещё подчиняется скоупам),
поэтому существующие ключи продолжают работать. Действуют обе проверки: ключ должен удовлетворять
требуемым скоупам эндпоинта **и** иметь право на ресурс. Ключ без права на процедуру получает HTTP 403.

## Аккаунты админов

- Админка защищена локальными аккаунтами. Пароли хранятся через PBKDF2 (SHA-256) и должны быть не
  короче 8 символов; та же политика применяется к bootstrap-аккаунту и к аккаунтам, создаваемым или
  изменяемым через API.
- Первый админ создаётся из `Weir:Admin:Username` / `Weir:Admin:Password` при старте, только когда
  админа ещё нет. Bootstrap-админ получает роль `Admin`.
- Вход выдаёт короткоживущий JWT (по умолчанию 30 минут) плюс долгоживущий refresh-токен, хранимые
  браузером. Задайте стабильный `Weir:Jwt:SigningKey`, чтобы сессии переживали рестарт; если не задан,
  генерируется эфемерный ключ на процесс и сессии сбрасываются при рестарте. В Production ключ должен
  быть не короче 32 символов.
- Refresh-токен хранится только как хеш, отзываем и ротируется при каждом использовании: админ-PWA
  обменивает его (`POST /admin/api/auth/refresh`) на новый access-токен, когда access-токен истекает, а
  использованный refresh-токен отзывается. Выход (`POST /admin/api/auth/logout`) и смена пароля отзывают
  refresh-токены; истёкшие и отозванные токены вычищаются в фоне.
- JWT отзываемы до истечения. Каждый токен несёт версию, снятую с аккаунта, и проверяется на каждом
  запросе: смена пароля админа или отключение аккаунта отклоняет его действующие токены при следующем
  запросе, а не оставляет их валидными до истечения.
- Неудачные входы ограничиваются по IP клиента (а не по логину), поэтому поток неверных паролей не
  заблокирует настоящего админа; при отсутствии аккаунта пароль сверяется с ложным хешем, чтобы время
  ответа не выдавало существующие логины. Состояние троттлинга хранится в control-plane, поэтому
  блокировка переживает перезапуск и общая для всех инстансов в HA-развёртывании.
- Дополнительные админы и смена паролей - в разделе **Admins**.

### Роли

У аккаунтов админов одна из двух ролей:

| Роль | Доступ |
| :-- | :-- |
| `Admin` | Полный доступ - читать всё и вносить изменения (эндпоинты, ключи, области, админы). |
| `Viewer` | Только чтение - дашборд, эндпоинты, ключи и аудит, без изменений. |

Каждый изменяющий маршрут admin API требует роль `Admin`; маршруты чтения доступны любому вошедшему
админу. `Viewer`, пытающийся внести изменение, получает HTTP 403.

## Персональные access-токены (admin API)

JWT сессии в браузере короткоживущий и не предназначен для скриптов. Для автоматизации - например,
задача CI/CD, синхронизирующая параметры эндпоинта при деплое - админ создаёт **персональный
access-токен** в разделе **Account -> Access tokens** (или через `POST /admin/api/account/tokens`).

- Токен выглядит как `weadm_<random>`. Хранятся только SHA-256 хеш и короткий префикс; открытый текст
  показывается один раз при создании и не сохраняется.
- Токен аутентифицирует как его владелец-админ, с текущей ролью этого админа. Если аккаунт отключён
  или роль изменилась, токен сразу это отражает. Токен `Viewer` даёт только чтение.
- Токены могут иметь необязательный срок действия и отзываются мгновенно (без кеша) в разделе
  **Account** или через `DELETE /admin/api/account/tokens/{id}`.
- Один админ может держать не более `Weir:Admin:MaxTokensPerAdmin` токенов (по умолчанию 20). Задайте
  `Weir:Admin:RequireTokenExpiry`, чтобы требовать у каждого токена срок действия.
- Передавайте его как `Authorization: Bearer weadm_...`. Любой маршрут admin API принимает и токен,
  и JWT входа.

Каждый админ управляет только своими токенами; id токена берётся из личности вызывающего, поэтому один
админ не может просматривать или отзывать токены другого.

Пример - обновить параметры хранимой процедуры из CI после миграции БД, пересинхронизировав из базы.
`POST /admin/api/endpoints/sync` сверяет метаданные параметров подходящих эндпоинтов с тем, что теперь
сообщает база; сузьте фильтрами в query (все объединяются по `AND`, регистронезависимо):

```sh
# WEIR_ADMIN_TOKEN - секрет в окружении CI (weadm_...).
AUTH="Authorization: Bearer $WEIR_ADMIN_TOKEN"

# Конкретная БД на конкретном сервере (именованное подключение):
curl -fsS -X POST "$WEIR_URL/admin/api/endpoints/sync?connection=sales" -H "$AUTH"

# Конкретная процедура (по имени, без id эндпоинта): connection + schema + object.
curl -fsS -X POST "$WEIR_URL/admin/api/endpoints/sync?connection=sales&schema=dbo&object=GetOrders" -H "$AUTH"

# Всё:
curl -fsS -X POST "$WEIR_URL/admin/api/endpoints/sync" -H "$AUTH"

# Или один эндпоинт по его id, если он уже известен:
curl -fsS -X POST "$WEIR_URL/admin/api/endpoints/$ENDPOINT_ID/sync" -H "$AUTH"
```

Ответ - JSON-массив с `status` по каждому эндпоинту (`updated`, `unchanged`, `objectNotFound` или
`error`). Синхронизация только сверяет уже существующие эндпоинты, новые не создаёт. Она возвращает
HTTP 200, даже если ничего не совпало (пустой массив), поэтому CI-задаче, ожидающей изменение, стоит
проверять результаты, а не только код статуса.

## Аудит и логирование безопасности

Аудируется каждая административная мутация: входы, создание/отзыв ключей и токенов, создание/изменение/
удаление/импорт эндпоинтов, upsert/удаление scope, создание админа и сброс пароля, изменения настроек -
с записью действующего админа и затронутого ресурса. Аудит вызовов data-plane опционален
(`Weir:Audit:DataPlane`). Журнал аудита виден в разделе **Audit** и запрашивается через
`GET /admin/api/audit`, и чистится фоновым сервисом по настройке рантайма `AuditRetentionDays`
(см. [Конфигурация](configuration.md#логирование)).

События безопасности - неудачные входы, блокировки, отказы по scope/grant и срабатывания rate-limit -
также пишутся в файловый лог (см. [Конфигурация](configuration.md#логирование)), каждый запрос несёт
correlation id.

## Усиление (hardening)

- Интроспекция (`/admin/api/introspect/*`), перечисляющая объекты БД, требует роль `Admin`; роут
  connection-health скрывает детали ошибок драйвера для не-админов (Viewer).
- Гарды data-plane идут с безопасными ненулевыми дефолтами: лимит строк, gateway-таймаут, лимит строк
  TVP и лимит размера тела (см. [Конфигурация](configuration.md)). Ключу без своего rate-limit можно
  задать дефолт через `Weir:DataPlane:DefaultApiKeyRateLimitPerMinute`.
- Значимые для безопасности опции валидируются на старте, поэтому конфиг-ошибка падает сразу.
- Поставляемая строка подключения по умолчанию не отключает проверку TLS-сертификата.

## Хорошие практики

- Задавайте надёжные уникальные значения `Weir:Admin:Password` и `Weir:Jwt:SigningKey` в проде.
- Держите строки подключения data-plane вне системы контроля версий; передавайте их через окружение
  или secret-хранилище.
- Разворачивайте Weir за TLS. Админка и admin API делят origin хоста (нет поверхности CORS).
- Значения параметров по умолчанию не логируются, поэтому телеметрия и аудит остаются PII-safe.
