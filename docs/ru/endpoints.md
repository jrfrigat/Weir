# Weir - Эндпоинты и контракт API

> [English](../en/endpoints.md) - [Начало работы](getting-started.md) - [Архитектура](architecture.md)

Эндпоинт - это метаданные, отображающие HTTP-маршрут на хранимую процедуру или функцию. Эндпоинты
управляются из админки (или admin API) и применяются сразу - без передеплоя.

## Поля эндпоинта

| Поле | Смысл |
| :-- | :-- |
| Route | Путь под `/api`, например `orders/create`. Может содержать захваты, например `orders/{id}` - см. ниже. Уникален в рамках HTTP-метода. |
| HttpMethod | GET / POST / PUT / PATCH / DELETE. |
| ConnectionName | Именованное подключение, на котором выполнять. |
| ObjectType | StoredProcedure, TableValuedFunction или ScalarFunction. |
| Schema / ObjectName | например `dbo` / `usp_CreateOrder`. |
| ResultMode | MultiRow, SingleRow, Scalar, NonQuery или MultiResultSet (информационно). |
| CommandTimeoutSeconds | Опциональный таймаут команды. |
| Enabled | Обслуживается ли эндпоинт. |
| Cache | Политика кеша результата (см. ниже). |
| Parameters | Определения параметров (см. ниже). |
| RequiredScopes | Скоупы, которыми должен обладать ключ. Пусто - любой аутентифицированный ключ. |

## Шаблоны маршрутов

Маршрут может захватывать сегмент целиком: `orders/{id}` совпадает с `GET /api/orders/123` и делает
`123` доступным любому параметру с `Source = Route` и `Name = id` (имя сопоставляется без учёта
регистра). Захват берёт ровно один сегмент, поэтому `orders/{id}` не совпадает с `orders/123/lines`
и никогда не совпадает с пустым сегментом.

Захват - это либо весь сегмент, либо ничего: `{id}` - захват, а `v{id}` - обычный литеральный
сегмент, в котором просто есть фигурные скобки.

Разрешение детерминировано. Литеральный маршрут всегда выигрывает у шаблона, который тоже подошёл бы,
поэтому `orders/count` и `orders/{id}` могут сосуществовать. Из двух шаблонов выигрывает тот, у
которого больше литеральных сегментов: `orders/{id}/lines` проверяется раньше `orders/{id}/{part}`.
Два маршрута, которые ни один запрос не различит - `orders/{id}` и `orders/{orderId}`, - это коллизия:
отдаётся более позднее определение, а шлюз пишет предупреждение при старте.

## Параметры

Каждый параметр объявляет, откуда берётся значение и как оно биндится к БД.

| Поле | Смысл |
| :-- | :-- |
| Name | Логическое имя - ключ в теле JSON или query, который использует клиент. |
| DbParameterName | Имя параметра в БД (по умолчанию Name). |
| Source | Body, Query, Route, Header, Claim или Const. |
| Direction | Input, Output, InputOutput или ReturnValue. |
| DbType | Провайдер-независимый тип; `Structured` - это table-valued параметр. |
| Required | Должен ли запрос его передать. |
| DefaultValue | Применяется, когда запрос опускает значение. |
| Size / Precision / Scale | Для размерных / числовых типов. |
| TypeName | Для TVP: имя SQL-типа, например `dbo.OrderItemType`. |
| TableColumns | Для TVP: упорядоченная схема колонок. |
| ValidationRegex | Опциональное регулярное выражение, которому должно соответствовать значение. |

## Запрос

Тело запроса - плоский JSON-объект, ключи которого - логические имена параметров. Скаляры - это
значения; table-valued параметр - массив объектов. Кеш и таймауты настраиваются на сервере, а не
передаются клиентом.

```http
POST /api/orders/create
X-Api-Key: wk_live_9f3a...
Content-Type: application/json

{
  "customerId": 42,
  "comment": "rush",
  "items": [
    { "sku": "A1", "qty": 2 },
    { "sku": "B2", "qty": 5 }
  ]
}
```

Для GET-эндпоинтов параметры берутся из query-строки:
`GET /api/customers/get?customerId=42`.

## Ответ

Каждый успешный ответ - единый конверт. `data` - это массив наборов строк (массив массивов);
`messages` несёт SQL `PRINT` / информационные сообщения.

```json
{
  "data": [
    [ { "id": 1001, "status": "created" } ]
  ],
  "output": { "newOrderId": 1001, "totalCount": 57 },
  "returnValue": 0,
  "rowsAffected": 1,
  "truncated": false,
  "messages": [
    { "text": "Order created", "severity": 0, "number": 50000, "procedure": "usp_CreateOrder", "line": 12 }
  ]
}
```

- Один набор строк - `"data": [ [ ... ] ]`; пустой набор - `"data": [ [] ]`.
- `output` содержит значения output / input-output параметров по логическому имени, либо null.
- `returnValue` - RETURN процедуры, либо null.
- `truncated` - `true`, когда результат упёрся в лимит `Weir:DataPlane:MaxRows` и был обрезан
  (иначе `false`).
- Ошибка SQL (severity >= 11 или `THROW`) превращается в HTTP 4xx/5xx `application/problem+json`
  (RFC 7807); низкоуровневые `PRINT` / info-сообщения попадают в `messages` при успехе.
- `messages` можно подавить на уровне эндпоинта. Включите **Suppress SQL messages** в редакторе эндпоинта
  (флаг `SuppressMessages`), и массив всегда будет пустым, чтобы болтливые диагностики процедуры не
  доходили до вызывающих. Свойство остаётся на месте, поэтому форма конверта не меняется.

## Table-valued параметры (TVP)

Объявите параметр с `DbType = Structured`, задайте `TypeName` (SQL-тип TVP) и определите
`TableColumns`. Клиент отправляет массив объектов; ключи сопоставляются с именами колонок.

## Output-параметры и return value

Объявите параметры с направлением `Output` или `InputOutput`; их значения появятся в `output` после
выполнения. Для хранимых процедур целочисленный RETURN всегда попадает в `returnValue`.

## Кеширование

Задайте политику кеша на эндпоинт: `Enabled`, `TtlSeconds`, `VaryByParameters` (входные параметры,
чьи значения формируют ключ кеша), `VaryByApiKey` и `CoalesceRequests`. Отрендеренный JSON кешируется
в памяти; клиент не может это обойти. Кеш уместнее всего для read-only эндпоинтов. Правка или удаление эндпоинта сразу
инвалидирует его закешированные ответы, поэтому изменение никогда не отдаёт устаревшие данные. Также
можно принудительно очистить кеш в любой момент - по эндпоинту из админ-панели или по фильтру через
admin API для CI/CD (см. [Очистка кеша](#очистка-кеша) ниже).

Кешируемые ответы на `GET` несут HTTP-валидаторы кеша: сильный `ETag`, вычисленный из точных байтов
ответа, и `Cache-Control: private, max-age=<TtlSeconds>`. Клиент, повторяющий запрос с
`If-None-Match: <etag>` (или `*`), получает `304 Not Modified` без тела, когда закешированный ответ не
изменился, экономя передачу. `304` отдаётся до того, как в поток пойдут байты. Усечённый ответ (тот,
что упёрся в лимит `MaxRows`) не кешируется и не несёт `ETag`, поэтому частичное тело никогда не
сверяется с полным.

### Одновременные вызовы одного ключа

`CoalesceRequests` (включено по умолчанию) определяет, что происходит, когда вызовы с одним ключом кеша
приходят, пока ответ для него уже готовится - на холодном ключе или в момент его истечения. Если
включено, первый вызов выполняет процедуру, а остальные ждут его байты: всплеск стоит одного обращения
к БД вместо одного на каждый вызов. Это ровно тот момент, когда кеш сам по себе помогает меньше всего,
потому что попасть в него ещё никто не может.

Запрос принадлежит всем, кто его ждёт, а не тому, кто пришёл первым. Отключение клиента или его
шлюзовой таймаут завершают только его собственный запрос - сам запрос к БД продолжает работать для тех,
кто стоит за ним в очереди. Он снимается, только когда ушёл последний ждущий, то есть живёт ровно
столько, сколько кому-то нужен ответ. Вызов, обслуженный таким образом, считается попаданием в кеш (он
не обращался к БД), а его длительность включает ожидание - именно ту задержку, которую увидел клиент.

Выключайте для эндпоинта, который обязан выполнять процедуру на каждый вызов даже при включённом кеше,
или как аварийный рычаг. Настройка имеет смысл только при включённом кеше: именно ключ кеша определяет,
что два вызова спрашивают одно и то же, - без него ждать нечего.

## Управление эндпоинтами через admin API

Эндпоинтами управляют в разделе **Endpoints** в админке или через admin API по адресу
`/admin/api/endpoints`. Вызовы admin API аутентифицируются JWT входа или персональным access-токеном
(`Authorization: Bearer ...`); о токенах и использовании в CI/CD см. [Безопасность](security.md).
Каждый изменяющий маршрут требует роль `Admin`.

| Действие | Маршрут |
| :-- | :-- |
| Список всех эндпоинтов | `GET /admin/api/endpoints` |
| Получить один | `GET /admin/api/endpoints/{id}` |
| Создать или обновить | `POST /admin/api/endpoints` / `PUT /admin/api/endpoints/{id}` |
| Удалить | `DELETE /admin/api/endpoints/{id}` |
| Импортировать набор (upsert по id) | `POST /admin/api/endpoints/import` |
| Выполнить один (admin "try it") | `POST /admin/api/endpoints/{id}/invoke` |
| Сгенерировать документ OpenAPI 3.0 | `GET /admin/api/openapi.json` |

Создание или обновление сразу перезагружает каталог в памяти; то, что задублировало бы существующие
метод + маршрут, возвращает HTTP 409.

### Синхронизация параметров из базы

После того как миграция изменила сигнатуру процедуры, пересинхронизируйте, чтобы определения
параметров эндпоинта совпали с тем, что теперь сообщает база. Синхронизация сверяет только метаданные
`Parameters`; она не меняет процедуру и обновляет только уже существующие эндпоинты (новые не создаёт).

| Область | Маршрут |
| :-- | :-- |
| Один эндпоинт по id | `POST /admin/api/endpoints/{id}/sync` |
| Конкретная БД на сервере (именованное подключение) | `POST /admin/api/endpoints/sync?connection=sales` |
| Целая схема | `POST /admin/api/endpoints/sync?connection=sales&schema=dbo` |
| Одна процедура по имени | `POST /admin/api/endpoints/sync?connection=sales&schema=dbo&object=GetOrders` |
| Все эндпоинты | `POST /admin/api/endpoints/sync` |

Фильтры `connection`, `schema` и `object` объединяются по AND и регистронезависимы. Массовый маршрут
возвращает HTTP 200 с JSON-массивом результатов по каждому эндпоинту, у каждого свой `status`:

| Status | Смысл |
| :-- | :-- |
| `updated` | Параметры изменились и сохранены. |
| `unchanged` | Уже синхронизировано; ничего не записано. |
| `objectNotFound` | Процедура эндпоинта не найдена на его подключении. |
| `error` | Синхронизация этого эндпоинта не удалась; причина в сообщении результата. |

Пустой массив означает, что под фильтр ничего не попало, поэтому CI/CD-задаче, ожидающей изменение,
стоит проверять результаты, а не только код статуса. Полный пример для CI/CD с персональным
access-токеном см. в [Безопасности](security.md).

### Очистка кеша

Кеш обычно истекает по своему TTL, а редактирование или удаление эндпоинта и так вытесняет его
закешированные ответы. Когда деплой меняет данные за закешированным чтением - миграция пересоздаёт
таблицу, задача переписывает справочные данные - выполните принудительную очистку, чтобы клиенты сразу
увидели новые данные, а не ждали истечения TTL. Очистка удаляет только уже отрендеренные ответы; она
никогда не меняет определение эндпоинта, и кеш наполняется заново при следующем вызове.

В админ-панели действие **Purge cache** в строке эндпоинта с включённым кешем очищает этот эндпоинт.
Через admin API (JWT входа или персональный access-токен, роль `Admin`) один маршрут покрывает и
отдельный эндпоинт, и удобную для пайплайна массовую инвалидацию:

| Область | Маршрут |
| :-- | :-- |
| Один эндпоинт по id | `POST /admin/api/endpoints/{id}/cache/purge` |
| Один маршрут | `POST /admin/api/cache/purge?route=orders/get` |
| Одна процедура по имени (по всем подключениям) | `POST /admin/api/cache/purge?object=GetOrders` |
| Конкретная БД на сервере (именованное подключение) | `POST /admin/api/cache/purge?connection=sales` |
| Целая схема | `POST /admin/api/cache/purge?connection=sales&schema=dbo` |
| Все эндпоинты на провайдере (коннекторе) | `POST /admin/api/cache/purge?provider=SqlServer` |
| Все эндпоинты | `POST /admin/api/cache/purge` |

Фильтры `route`, `connection`, `schema`, `object` и `provider` объединяются по AND и регистронезависимы;
`provider` сопоставляется с коннектором за подключением эндпоинта (например `SqlServer` или
`PostgreSql`). Без фильтра очищается кеш всех эндпоинтов. Оба маршрута возвращают HTTP 200 с
`{ "matchedEndpoints": <n>, "purgedRoutes": [ ... ] }`, где `purgedRoutes` перечисляет очищенные
уникальные маршруты. Каждая очистка записывается в журнал аудита под категорией `cache.purge`.
